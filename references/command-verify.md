# verify 命令参考

## 命令概述
`verify` 命令用于运行验证循环，确保代码或系统的质量和可靠性，帮助用户发现和修复潜在问题。

## 命令语法
```bash
fullstack-dev-skills verify [选项] --target <目标>
```

## 参数说明
| 参数 | 类型 | 必需 | 描述 |
|------|------|------|------|
| `--target` | 字符串 | 是 | 验证目标，可以是文件路径、目录路径或系统组件名称 |
| `--tests` | 字符串数组 | 否 | 要运行的测试类型，支持 `unit`、`integration`、`e2e`、`security` 等 |
| `--criteria` | 字符串数组 | 否 | 验证标准，支持 `quality`、`performance`、`security`、`maintainability` 等 |
| `--timeout` | 数字 | 否 | 验证超时时间（秒），默认 600 |
| `--report` | 字符串 | 否 | 报告格式，支持 `text`（默认）、`json` 和 `html` |
| `--output` | 字符串 | 否 | 报告输出路径，默认输出到控制台 |

## 工作原理
1. **目标分析**：分析验证目标的类型和结构
2. **测试选择**：根据目标类型和参数选择合适的测试
3. **执行验证**：运行选定的测试和验证标准
4. **结果分析**：分析验证结果，识别问题和改进机会
5. **报告生成**：生成详细的验证报告
6. **建议提供**：基于验证结果提供改进建议

## 常见问题

### Q: verify 命令可以验证哪些类型的目标？
A: `verify` 命令可以验证以下类型的目标：
- **单个文件**：JavaScript、TypeScript、Python、Java 等代码文件
- **目录**：包含多个代码文件的目录
- **项目**：完整的项目目录结构
- **系统组件**：特定的系统组件或服务

### Q: 支持哪些测试类型？
A: `verify` 命令支持以下测试类型：
- **unit**：单元测试，验证单个函数或模块
- **integration**：集成测试，验证多个模块或服务的交互
- **e2e**：端到端测试，验证完整的用户流程
- **security**：安全测试，验证潜在的安全漏洞
- **performance**：性能测试，验证系统的性能指标

### Q: 支持哪些验证标准？
A: `verify` 命令支持以下验证标准：
- **quality**：代码质量，包括复杂度、命名、格式等
- **performance**：性能，包括响应时间、资源使用等
- **security**：安全性，包括漏洞、注入风险等
- **maintainability**：可维护性，包括代码结构、注释等
- **reliability**：可靠性，包括错误处理、边界情况等

### Q: 验证过程会修改代码吗？
A: 验证过程是只读操作，不会修改任何代码或文件。它只会分析代码并生成报告。

### Q: 如何选择合适的测试和验证标准？
A: 选择测试和验证标准时考虑以下因素：
- **目标类型**：不同类型的目标需要不同的测试
- **验证目的**：根据验证的目的选择合适的标准
- **时间限制**：更全面的验证需要更多时间
- **项目阶段**：不同项目阶段的验证重点不同

### Q: 验证失败的原因有哪些？
A: 验证失败的常见原因包括：
- **代码质量问题**：复杂度高、命名不当、重复代码等
- **测试失败**：单元测试、集成测试等失败
- **性能问题**：响应时间长、资源使用高等
- **安全漏洞**：XSS、SQL 注入等安全问题
- **配置问题**：环境配置、依赖问题等

### Q: 如何解读验证报告？
A: 解读验证报告时关注以下几点：
1. **总体分数**：了解整体质量水平
2. **关键问题**：优先处理严重的问题
3. **改进建议**：按照建议进行优化
4. **测试覆盖率**：确保测试覆盖足够的代码
5. **趋势分析**：与之前的验证结果比较，了解改进情况

### Q: 验证报告中的分数是如何计算的？
A: 验证报告中的分数基于以下因素计算：
- 测试通过率
- 代码质量指标
- 性能指标
- 安全漏洞数量
- 代码覆盖率

不同验证标准的权重可能不同，具体计算方法会根据验证目标和类型进行调整。

### Q: 如何使用验证报告改进代码？
A: 使用验证报告改进代码的方法包括：
1. **优先级排序**：按照问题的严重程度排序
2. **分类处理**：将问题分类，批量处理类似问题
3. **持续验证**：修复后再次运行验证，确保问题已解决
4. **预防措施**：实施编码规范和工具，防止类似问题再次出现
5. **团队共享**：与团队共享验证报告，提高整体代码质量

### Q: 定期运行验证命令有什么好处？
A: 定期运行验证命令的好处包括：
- **及早发现问题**：在问题扩大前发现并解决
- **保持代码质量**：持续监控代码质量指标
- **防止回归**：确保修复不会引入新问题
- **提高安全性**：定期检查安全漏洞
- **优化性能**：持续监控和优化性能指标
