# 上下文研究模式参考

研究模式上下文的核心内容、探索策略、深度分析配置和知识整理方法，用于构建系统化、高效的知识获取与整理流程。

## When to Activate

- 需要深入了解代码库结构
- 进行技术调研与方案评估
- 整理项目知识文档
- 分析复杂系统架构
- 探索未知技术领域

## Core Principles

### 1. 广度优先，深度渐进

研究应先建立全局视角，再逐步深入细节。

```markdown
# 研究层次结构

## 第一层：概览扫描
- 项目结构总览
- 核心模块识别
- 技术栈确认

## 第二层：模块分析
- 模块职责划分
- 接口定义梳理
- 依赖关系映射

## 第三层：实现细节
- 核心算法分析
- 数据流向追踪
- 关键路径识别
```

### 2. 结构化知识输出

研究结果应转化为可复用的知识资产。

```markdown
# 知识输出模板

## 概述
[主题的简要描述和核心价值]

## 关键发现
- 发现点 1：[描述]
- 发现点 2：[描述]
- 发现点 3：[描述]

## 技术细节
[深入的技术分析和代码示例]

## 最佳实践
[基于研究得出的建议]

## 参考资料
[相关文档和资源链接]
```

### 3. 迭代深化策略

研究过程应支持多轮迭代，逐步完善认知。

```markdown
# 迭代研究流程

## 初始探索
目标：建立基础认知
方法：快速扫描、关键词搜索
输出：概念图谱初稿

## 深度分析
目标：理解核心机制
方法：代码阅读、文档查阅
输出：详细分析报告

## 验证确认
目标：确保理解准确
方法：测试验证、专家确认
输出：最终知识文档
```

## 深度分析配置

### 代码分析配置

```yaml
# 深度分析配置示例

analysis:
  scope:
    - type: "source"
      paths: ["src/", "lib/"]
      exclude: ["node_modules", "dist", "build"]
    
    - type: "config"
      paths: ["config/", "*.config.js", "*.json"]
    
    - type: "test"
      paths: ["tests/", "test/", "__tests__/"]

  depth:
    level: "deep"
    max_files: 100
    max_lines_per_file: 500

  focus_areas:
    - name: "核心业务逻辑"
      patterns: ["service", "handler", "controller"]
    
    - name: "数据访问层"
      patterns: ["repository", "dao", "model"]
    
    - name: "接口定义"
      patterns: ["api", "route", "endpoint"]
```

## 知识整理方法

### 分类整理法

```markdown
# 知识分类体系

## 按技术层次分类

- 数据结构算法
- 设计模式

### 框架层
- 框架核心概念
- 最佳实践
- 常见问题

### 应用层
- 业务逻辑实现
- 集成方案
- 性能优化

## 按知识类型分类
### 概念知识
- 定义和原理
- 适用场景
- 优缺点分析

### 实践知识
- 代码示例
- 配置模板
- 调试技巧

### 问题知识
- 常见错误
- 排查方法
- 解决方案
```

```markdown

## 横向关联
```

## 探索策略

### 广度优先探索

```markdown
# 广度优先探索流程

## 阶段 1：快速扫描
时间：总时间的 20%
目标：建立全局认知
方法：
  - 目录结构浏览
  - README 阅读
  - 配置文件检查

## 阶段 2：重点识别
时间：总时间的 30%
目标：识别关键模块
方法：
  - 入口文件分析
  - 核心模块定位
  - 依赖关系梳理

## 阶段 3：深度分析
时间：总时间的 40%
目标：理解核心机制
方法：
  - 代码详细阅读
  - 流程追踪分析
  - 文档对照学习

## 阶段 4：知识整理
时间：总时间的 10%
目标：输出知识资产
方法：
  - 笔记整理
  - 图表绘制
  - 文档编写
```

### 深度优先探索

```markdown
# 深度优先探索流程

## 选择切入点
原则：
  - 从核心功能入口开始

方法：
  - 调用链路追踪

## 循环迭代
条件：
  - 达到预定深度
  - 完成目标覆盖
```

## Quick Reference: 研究模式速查

| 策略 | 适用场景 | 时间分配 |
|------|----------|----------|
| 广度优先 | 新项目探索 | 20% 扫描 + 30% 识别 + 40% 分析 + 10% 整理 |
| 深度优先 | 问题排查 | 切入点选择 + 链路追踪 + 回溯扩展 |
| 问题驱动 | Bug 修复 | 问题定义 + 假设生成 + 验证执行 |
| 对比分析 | 技术选型 | 特性对比 + 场景匹配 + 权衡决策 |

## Anti-Patterns to Avoid

```markdown
# 研究反模式

## 错误：无目标漫游
表现：漫无目的地浏览代码
正确：明确研究目标，有针对性地探索

## 错误：过早深入细节
表现：一开始就陷入具体实现
正确：先建立全局认知，再深入细节

## 错误：忽视文档
表现：只看代码不看文档
正确：代码与文档结合，相互印证

## 错误：知识不整理
表现：研究完不做记录和总结
正确：及时整理输出，形成知识资产

## 错误：单一信息源
表现：只依赖一个信息来源
正确：多渠道验证，交叉确认

## 错误：忽略版本差异
表现：参考过时的文档或代码
正确：确认版本一致性，注意变更历史
```

**Remember**: 有效的研究需要明确的目标、系统的方法和结构化的输出。研究不仅是获取信息，更是将信息转化为可复用的知识资产。
