# 代码审查工作流参考

代码审查的核心流程、审查清单、反馈模板和最佳实践，用于确保代码质量、安全性和可维护性。

## When to Activate

- 提交代码审查请求时
- 审查他人代码时
- 进行代码质量评估时
- 制定团队审查规范时

## Core Principles

### 1. 尊重与建设性

审查应聚焦于代码本身，而非开发者个人。反馈应当具体、可操作且富有建设性。

```markdown
# Good: 具体且建设性的反馈
建议将 `processData` 函数拆分为更小的函数。
当前函数有 150 行，包含多个职责，违反单一职责原则。
可以考虑：
- `validateInput()` - 处理输入验证
- `transformData()` - 处理数据转换
- `persistResult()` - 处理结果存储

# Bad: 模糊或攻击性的反馈
这段代码写得不好，需要重写。
```

### 2. 及时性与效率

审查应及时完成，避免阻塞开发进度。同时保持足够的深度。

```markdown
# Good: 明确时间预期
我会在 24 小时内完成此审查。
优先级：P0（阻塞发布）会在 4 小时内响应。

# Bad: 无限期延迟
我稍后看看...
```

### 3. 一致性与标准化

使用统一的审查标准和模板，确保团队审查质量一致。

## 审查清单

### 代码风格审查

```markdown
## 代码风格检查项

### 命名规范
- [ ] 变量名使用有意义的描述性名称
- [ ] 函数名使用动词或动词短语（如 `getUserById`）
- [ ] 类名使用名词或名词短语（如 `UserRepository`）
- [ ] 常量使用全大写下划线命名（如 `MAX_RETRY_COUNT`）
- [ ] 避免使用缩写或单字母变量（循环变量除外）

### 格式规范
- [ ] 缩进一致（空格或 Tab，不混用）
- [ ] 行长度不超过限制（通常 80-120 字符）
- [ ] 适当使用空行分隔逻辑块
- [ ] 大括号风格一致（K&R 或 Allman）
- [ ] 逗号后有空格，逗号前无空格

### 注释规范
- [ ] 复杂逻辑有解释性注释
- [ ] 公共 API 有文档注释
- [ ] 注释与代码保持同步更新
- [ ] 避免无意义的注释（如 `// i++`）
- [ ] TODO 注释有责任人或工单号

### 代码组织
- [ ] 文件结构清晰（导入、常量、类、函数）
- [ ] 导入语句按规范排序
- [ ] 无未使用的导入和变量
- [ ] 函数长度适中（通常 < 50 行）
- [ ] 类的职责单一
```

### 逻辑审查

## 逻辑检查项

### 业务逻辑
- [ ] 逻辑正确性：代码实现了预期功能

- [ ] 避免深层嵌套（建议 < 4 层）
- [ ] 提前返回减少嵌套
- [ ] switch/case 有 default 分支

### 数据处理
- [ ] 数据类型正确
- [ ] 数据精度满足要求
- [ ] 时区处理正确
- [ ] 字符串编码一致

### 并发处理
- [ ] 线程安全考虑
- [ ] 锁的正确使用
- [ ] 避免死锁风险
- [ ] 资源竞争处理
- [ ] 异步操作正确性

```markdown
## 安全检查项

### 输入验证

## 反馈模板示例

### 示例 1：代码风格问题

```markdown
### 命名不规范

**文件**: `src/services/userService.ts:42`
**严重程度**: 建议

**问题描述**
变量名 `u` 不够描述性，影响代码可读性。

**当前代码**
```typescript
const u = await db.users.findById(userId);

**建议修改**
```typescript
const user = await db.users.findById(userId);
return user;
```

**修改原因**
使用完整的变量名 `user` 可以提高代码可读性，特别是在较长的函数中。
遵循团队代码规范中的命名约定。

### 示例 2：安全问题

```markdown
### SQL 注入风险

**文件**: `src/api/routes/user.ts:78`
**严重程度**: 阻塞

**问题描述**
直接拼接用户输入到 SQL 查询中，存在 SQL 注入风险。

**当前代码**
```typescript
const query = `SELECT * FROM users WHERE id = '${userId}'`;
const result = await db.execute(query);
```

**建议修改**
```typescript
const query = 'SELECT * FROM users WHERE id = ?';
const result = await db.execute(query, [userId]);
```

**修改原因**
使用参数化查询可以防止 SQL 注入攻击。
用户输入 `userId` 可能包含恶意 SQL 代码，直接拼接会导致安全漏洞。

**参考资料**
- [OWASP SQL Injection Prevention](https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html)
```

### 示例 3：性能问题

```markdown
### N+1 查询问题

**文件**: `src/services/orderService.ts:156`
**严重程度**: 重要

**问题描述**
在循环中执行数据库查询，导致 N+1 查询问题，影响性能。

**当前代码**
```typescript
const orders = await db.orders.findAll();
for (const order of orders) {
    order.user = await db.users.findById(order.userId);
}
return orders;
```

**建议修改**
```typescript
const orders = await db.orders.findAll();
const userIds = [...new Set(orders.map(o => o.userId))];
const users = await db.users.findByIds(userIds);
const userMap = new Map(users.map(u => [u.id, u]));

return orders.map(order => ({
    ...order,
    user: userMap.get(order.userId)
}));
```

**修改原因**
原代码对每个订单都执行一次用户查询，如果有 100 个订单，
就会执行 1 + 100 = 101 次数据库查询。
优化后只需 2 次查询，大幅提升性能。

**性能对比**
- 原方案: 1 + N 次查询
- 优化后: 2 次查询
```

### 示例 4：逻辑问题

```markdown
### 边界条件处理缺失

**文件**: `src/utils/arrayHelper.ts:23`
**严重程度**: 阻塞

**问题描述**
函数未处理空数组的情况，可能导致运行时错误。

**当前代码**
```typescript
function getFirstItem<T>(items: T[]): T {
    return items[0];
}
```

**建议修改**
```typescript
/**
 * 获取数组第一个元素
 * @param items - 输入数组
 * @returns 第一个元素，如果数组为空则返回 undefined
 */
function getFirstItem<T>(items: T[]): T | undefined {
    if (items.length === 0) {
        return undefined;
    }
    return items[0];
}
```

**修改原因**
当传入空数组时，原代码返回 `undefined`，但类型签名表示返回 `T`，
这会导致类型不一致。明确处理边界条件并更新类型签名，
使函数行为更加可预测。

**测试用例建议**
```typescript
describe('getFirstItem', () => {
    it('should return first item', () => {
        expect(getFirstItem([1, 2, 3])).toBe(1);
    });
    
    it('should return undefined for empty array', () => {
        expect(getFirstItem([])).toBeUndefined();
    });
});
```
```

### 示例 5：正面反馈

```markdown
### 良好的错误处理实践

**文件**: `src/services/paymentService.ts:89`
**严重程度**: 良好实践

**问题描述**
这段代码展示了良好的错误处理模式，值得团队学习。

**代码示例**
```typescript
async function processPayment(orderId: string): Promise<PaymentResult> {
    try {
        const order = await orderRepository.findById(orderId);
        if (!order) {
            throw new OrderNotFoundError(orderId);
        }
        
        const result = await paymentGateway.charge(order);
        await orderRepository.updateStatus(orderId, 'paid');
        
        return result;
    } catch (error) {
        if (error instanceof OrderNotFoundError) {
            logger.warn('Order not found', { orderId });
            throw error;
        }
        
        logger.error('Payment processing failed', { 
            orderId, 
            error: error.message 
        });
        throw new PaymentProcessingError(orderId, error);
    }
}
```

**值得肯定的地方**
1. 使用自定义异常类型，便于错误分类处理
2. 错误日志包含上下文信息，便于问题排查
3. 区分业务异常和系统异常
4. 保持错误链，不丢失原始错误信息

**建议**
可以将此模式作为团队错误处理的标准模板。
```

```markdown
## 改进建议优先级

## Anti-Patterns to Avoid

```markdown
## 审查反模式

# Bad: 模糊的反馈
代码写得不好，需要改。

# Good: 具体的反馈
`processData` 函数有 150 行，建议拆分为：
- `validateInput()` - 输入验证
- `transformData()` - 数据转换

---

# Bad: 人身攻击
你写的代码太烂了。

# Good: 聚焦代码
这段代码存在 SQL 注入风险，建议使用参数化查询。

---

# Bad: 无理由拒绝
我不喜欢这种写法。

# Good: 有理有据
使用 `async/await` 替代回调地狱，可以提高代码可读性。
参考：[团队代码规范 - 异步处理](link)

---

# Bad: 过度挑剔
这个变量名少了一个字母，不够完美。

# Good: 抓住重点
聚焦于影响功能和质量的问题，风格问题可以自动化处理。

---

# Bad: 延迟审查
我太忙了，下周再看。

# Good: 及时响应
